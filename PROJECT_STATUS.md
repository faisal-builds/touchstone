## What works end to end

A new user can, **through the SDK or raw HTTP**, with no manual database access:

1. `signup` → receive a JWT scoped to a newly created organization.
2. `login` → re-authenticate and receive a JWT.
3. Create an API key (machine credential, returned once).
4. Create a workspace and project.
5. Register a verifier (code / model / process / hybrid), auto-versioned.
6. Submit an artifact for verification (`202`, status `pending`).
7. Poll until the verification-engine grades it and returns
   score / uncertainty / pass-fail.
8. The risk-engine scores the verification (risk band) and writes it back.
9. The audit-engine records every step into a per-org tamper-evident hash chain,
   exportable and verifiable via the audit CLI.
10. The reward-hacking-detector attacks a registered verifier with adversarial
    artifacts and scores its robustness, growing a corpus of discovered exploits.

Steps 1–6 are exercised live by the SDK integration test; step 7 (grading) is
exercised by the verification-engine integration test. The `scripts/demo.py`
script runs 1–7 against a live stack.

## Component detail

### `libs/touchstone-events` — ✅
Immutable, org-scoped `EventEnvelope` + payloads (requested/completed, risk,
reward-hack, audit) with topic routing. Single source of truth for the wire
format. Frozen.

### `services/control-plane` — ✅
- Config (pydantic-settings, `TOUCHSTONE_` prefix), structured logging, OTel hooks.
- Security: Argon2id API keys (`tsk_…`, base62, hashed) + JWT + user passwords;
  constant-time login.
- RBAC: org→workspace→project, roles OWNER/ADMIN/MEMBER/VIEWER/SERVICE.
- Data model: orgs, users, memberships, api_keys, workspaces, projects,
  verifiers (auto-versioned, `robustness_score`), verification_runs. Native
  enums persist `.value`. Alembic migration autogenerated against real PG.
- API: `/v1/auth/{signup,login}`, workspaces, projects, api-keys, verifiers,
  verifications (submit emits `verification.requested`), audit (read-only via the
  cross-database `AuditReader`), health, `/metrics`. Internal-only:
  `/v1/internal/auth/introspect` validates `tsk_` API keys for other services
  (service-token protected) so they need not read the `api_keys` table.
- Middleware: request-id + structured access logs + Prometheus; Redis
  token-bucket rate limiting (fail-open); **security headers** (HSTS, strict CSP,
  nosniff, frame-deny, COOP/CORP, Permissions-Policy), docs-exempt CSP.
- Tests: 36 (RBAC, crypto, cross-tenant isolation, auth flow, JWT→key, security
  headers).

### `services/verification-engine` — ✅
- Subprocess **sandbox** baseline (rlimits: CPU/mem/procs/files; network isolation
  via `unshare`; isolated interpreter; hard timeout). Isolation verified by tests
  that trigger it (infinite loop, memory bomb).
- **Hardened sandbox backends** (ADR-002): a stable `Sandbox` protocol with a
  config-selectable **gVisor (`runsc`) / Firecracker** OCI backend — a locked-down
  container (no network, read-only root, all caps dropped, unprivileged user,
  pid/memory/cpu limits + ulimits) running the same harness. A `build_sandbox`
  factory selects the backend; preflight fails loudly if the runtime is missing
  (no silent downgrade in prod). Command construction + preflight + fallback are
  unit-tested.
- Verifiers: code (sandbox), model (LLM-as-judge, mock + Anthropic providers),
  process (step supervision), hybrid (ensemble with disagreement→escalation).
- Worker: consumes `verification.requested`, executes, persists result, emits
  `verification.completed`; unparseable events are routed to a dead-letter topic.
  Repository via SQLAlchemy Core (decoupled from the control-plane ORM).
- **Artifact store**: one interface, two backends — `file://` (dev/CI) and
  `s3://` (production, lazy `boto3`, blocking calls offloaded to a thread). Bare
  keys resolve under the configured prefix; full `s3://` refs are honored
  verbatim; traversal is rejected.
- Tests: 37 (sandbox isolation, the gVisor/Firecracker backend selection +
  command lockdown, the file + S3 artifact backends via moto, each verifier
  family, ensemble, end-to-end worker).

### `sdks/python` — ✅
- Installable `touchstone-sdk`. Typed pydantic models, full exception hierarchy
  mapped from `problem+json`, sync `TouchstoneClient` covering the whole surface
  incl. `wait_for_verification` polling. README with usage example.
- Tests: 12 (10 unit via mock transport; 2 live e2e against in-process server + PG).

### `services/risk-engine` — ✅
- Consumes `verification.completed`; a deterministic, explainable risk model
  combines (1 − score) and uncertainty into a risk score + band
  (low/medium/high/critical), with a "failed runs are never low-risk" floor.
- Writes `risk_score` back to the run; emits `risk.scored`.
- Tests: 12 (9 scorer: monotonicity, band partition, fail-floor, weight
  validation, clamping; 3 worker end-to-end against PG).

### `services/audit-engine` — ✅
- Per-organization, tamper-evident **SHA-256 hash chain** (ADR-011): each record
  commits to the previous; edits/deletes/reordering break the chain and are
  detected by recomputation.
- Records all seven audited events (signup, login, api_key.created,
  verifier.registered, verification.requested, verification.completed,
  risk.scored). Per-org advisory-lock serialization; idempotent on redelivery.
- Per-org export + verify via `python -m touchstone_audit.cli {export,verify}`.
- **Owns its database (per-service split)**: `audit_records` lives in the
  audit-engine's own schema with its own Alembic migrations and is FK-free, so it
  can run on a fully isolated database. The audit-engine reads nothing from the
  control-plane; the control-plane reads the log read-only via `AuditReader`
  (`TOUCHSTONE_AUDIT_DATABASE_URL`) to serve `/v1/audit`.
- Tests: 10 (7 chain: determinism, prev-hash dependence, tamper + reorder
  detection; 3 worker: full 7-event chain build + verify, idempotency,
  DB-tamper detection).

### `services/reward-hacking-detector` — ✅ (the core IP)
- Measures how robust a verifier is against manipulation. Generates diverse
  adversarial artifacts that should not pass, runs them through the verifier
  (reusing the verification-engine's sandbox — no re-implementation), detects
  which passes were reward hacks, and scores robustness = 1 − exploit_rate with
  a **Wilson confidence interval**.
- Six attack strategies across six exploit categories: content corruption,
  judge manipulation (prompt injection), length bias, formatting facades,
  edge cases, and a model-generated adversary. Generation is seeded and
  deterministically replayable.
- Knowledge base: every exploit persisted and deduplicated per verifier
  (occurrences increment), a growing corpus of verifier failures, each linked to
  the verifier version it was found against; the corpus is searchable by
  verifier / version / category / severity / strategy / min-score / free text.
  Robustness written back onto the verifier row.
- Each exploit records *why* the verifier failed — a failure reason grounded in
  the verifier's own score and breakdown at the moment it was fooled.
- Robustness scoring reports both a plain rate (with Wilson confidence interval)
  and a **severity-weighted** score (critical exploits penalized more than
  minor ones).
- Version comparison, regression detection (statistically meaningful drop, not
  noise), and trend direction.
- HTTP API (launch / query / list / exploits / search / trend / compare /
  report), authenticated by **both API keys and the control-plane's user JWT**
  (the dashboard's credential) and tenant-isolated; background worker with
  retry, failure recovery (scans and re-runs stranded jobs faithfully — seed and
  seed cases are persisted at launch), auto-evaluation on `verifier.registered`,
  and a `reward_hacking.robustness_evaluated` event on completion.
- **Owns its database (per-service split)**: `robustness_evaluations`, `exploits`,
  and a `verifier_refs` replica live in RHD's own schema with its own Alembic
  migrations. RHD never reads or writes the control-plane database for verifier
  data: the replica is populated from the `verifier.registered` event, and the
  headline `robustness_score` is handed back via the `robustness_evaluated` event
  for the control-plane to apply (single-writer).
- **Federated auth**: `tsk_` API keys are validated by calling the control-plane's
  introspection endpoint (`HttpIntrospector`, authenticated with a short-lived
  service token, positive results cached briefly), so RHD reads no control-plane
  tables and can run on a fully isolated database. User JWTs are still verified
  locally with the shared secret.
- **Sandbox wiring**: attacks execute verifier code through a configurable
  `Sandbox` backend. The orchestrator accepts any backend; production (API app +
  worker) builds the configured gVisor/Firecracker backend from
  `TOUCHSTONE_RHD_SANDBOX_*` settings, mirroring the verification engine. Tests
  default to the subprocess baseline.
- Tests: 50 (30 unit: attack determinism, detection/dedup + failure reason,
  Wilson/regression + severity weighting, sandbox backend selection + wiring;
  20 integration against real sandbox + PG: weak-vs-strong discrimination,
  reproducibility, persistence + dedup + writeback, weighted-score + version
  linkage, corpus search, failure recovery, retry-to-failed, JWT acceptance +
  isolation, the full authenticated API flow with tenant isolation, and
  end-to-end auth federation against the real control-plane).

### `services/ivp` — Inline Verification Plane — ✅ (the inline tier; see ADR-015)
The inline/critical-path gateway: synchronous **allow/block/redact/escalate** on
live AI traffic within a latency budget. Independent service, no database of its
own; depends only on the control-plane (introspection) and the event bus.
- **Policy + routing** (`policy.py`): resolves the tenant policy from an
  epoch-versioned cache; **routes around** verifiers whose RHD robustness score is
  below the policy floor and robustness-weights the rest — Touchstone is the only
  system that can, because only it measures gameability.
- **Tiered execution** (`execution.py`): fast tier runs code verifiers in the
  verification-engine sandbox (tight CPU/mem/wall limits) with a TTL/LRU dedup
  cache; slow (model/process) verifiers are escalated, never run inline.
- **Decisioning** (`decision.py`): robustness-weighted aggregation →
  `touchstone_risk.RiskModel` → action; applies redaction; fail-closed on a
  critical verifier error. Streaming (`streaming.py`) early-exits on block/redact.
- **Resilience** (`resilience.py`): bulkhead backpressure, circuit breaker,
  latency budget, per-policy fail-open/closed — the mechanisms behind the
  multi-region/SLO posture (built, not yet live-hardened).
- **Auth** reuses control-plane introspection (no control-plane DB reads, like the
  RHD). **Telemetry**: Prometheus decision-latency SLO + action/tier/cache/degrade.
- **Event loops**: `inline.decision` → audit chain (compliance), `inline.escalated`
  → verification-engine (deep verdict), `inline.evasion_observed` → RHD
  (re-evaluation flywheel). All emission is off the hot path.
- **SDK middleware**: `InlineGuard` in both Python and TypeScript.
- 27 tests (policy/decision/resilience/execution unit + full gateway integration
  through the real sandbox). The inline consumers are covered in the audit (chain),
  verification-engine (escalation), and RHD (evasion) suites.

## What does NOT yet exist

- **Per-service-DB / single-writer split — *complete*.** Two services own their
  data in their **own** databases with their **own** Alembic migrations, fed
  entirely by events, and the cross-service couplings are all resolved:
  - The reward-hacking detector owns `robustness_evaluations`, `exploits`, and a
    `verifier_refs` replica; the control-plane is the **sole writer** of
    `verifiers.robustness_score` via a Kafka consumer.
  - The audit-engine owns `audit_records`; the control-plane reads it through a
    read-only, configurable cross-database connection (`AuditReader`).
  - **Auth is federated**: RHD validates `tsk_` API keys by calling the
    control-plane's introspection endpoint (with a short-lived service token),
    reading no control-plane tables. Both the RHD and audit databases can now be
    **fully isolated**. *(Verification-engine and risk-engine still share the
    control-plane database by design — they read/write control-plane-owned
    `verification_runs`/`verifiers`; that is intentional, not a coupling defect.)*
- Load/perf testing of the verification hot path — **harness built** (Locust suite
  under `load-tests/`, smoke/local/staging/stress profiles, configurable
  thresholds, `perf-smoke` CI job). It runs against the control-plane API today;
  **production-representative numbers (end-to-end completion, worker throughput,
  queue backlog) still require the live stack** — broker + worker behind the API,
  i.e. the staging/AWS validation run.
- Real model-provider grading in CI (mock provider is used; the Anthropic
  provider is wired but unexercised without a key).
- Live execution of the gVisor/Firecracker backends and the multi-service event
  flow over a running broker (no container runtime in this build sandbox).

## Honest scope note

This is a tested engineering **foundation**, not a deployable product. The
differentiating problems of an AI verification company — whether verification
genuinely resists reward hacking, and whether labs trust an independent grader —
are empirical and organizational, and live largely outside this codebase. The
reward-hacking detector is the only remaining subsystem that is differentiated
rather than standard infrastructure.
